## Tools

List every tool your agent will use. For each tool, fill in all four fields.
You must have at least 3 tools. The three required tools are listed — add any additional tools below them.

> **Signature rule (important):** The three required tools keep their locked interfaces

> and `create_fit_card(outfit, new_item)`. All stretch parameters are **appended after**

### Tool 1: search_listings

**What it does:**
Searches the static mock listings dataset for items matching the user's criteria (description keywords, max price, and size) and returns the matching listings.

**Input parameters:**
- `description` (str): Keyword description of the item the user is looking for.
- `size` (str | None): The user's requested size (e.g., "M", "XXS"). Optional.
- `max_price` (float | None): The maximum price the user is willing to pay. Optional.

**What it returns:**
A list of dictionaries `list[dict]`, where each dictionary represents a matched listing.

**What happens if it fails or returns nothing:**
If no listings match, it returns an empty list `[]`. The planning loop will catch this and initiate **Retry Logic with Fallback** (e.g., dropping the size constraint and trying again). If the retry also fails, it early-returns an error to the user.

---

### Tool 2: suggest_outfit

**What it does:**
Uses the Groq LLM to suggest one or more complete outfit combinations by pairing the newly found item with existing pieces from the user's wardrobe, influenced by current trends.

**Input parameters:**
- `new_item` (dict): The selected listing dictionary returned from `search_listings`.
- `wardrobe` (dict): The user's current wardrobe dictionary.
- `trends` (str | None = None): *(Stretch)* Optional trend context from `check_trends`. Appended after the required params.
- `style_profile` (str | None = None): *(Stretch)* Optional saved style-preference text loaded from `data/style_profile.json`, merged into the prompt as additional wardrobe context. Appended after the required params.

**What it returns:**
A string `str` containing the LLM-generated outfit suggestion.

**What happens if it fails or returns nothing:**
If the wardrobe is empty, the tool will rely on the LLM to suggest general pairing advice instead of relying on specific wardrobe pieces. If `trends` or `style_profile` are `None`, the prompt simply omits that context — the tool still produces a full suggestion.

---

### Tool 3: create_fit_card

**What it does:**
Uses the Groq LLM to generate a short, shareable social-media-style caption ("fit card") describing the complete outfit.

**Input parameters:**
- `outfit` (str): The outfit suggestion generated by the `suggest_outfit` tool.
- `new_item` (dict): The listing dictionary of the newly found item.
- `price_assessment` (str | None = None): *(Stretch)* Optional price-fairness note from `compare_price`, woven into the caption when present. Appended after the required params.

**What it returns:**
A string `str` containing the generated caption.

**What happens if it fails or returns nothing:**
If `outfit` is empty or whitespace-only, the tool generates a caption focused on the `new_item` alone (it does not raise). If `price_assessment` is `None`, the caption simply omits the price note.

### Additional Tools (if any)

### Tool 4: compare_price (Stretch Feature)

**What it does:**
Estimates whether the price of the selected item is fair based on comparable listings (same `category`, optionally narrowed by overlapping `style_tags`) in the dataset.

**Input parameters:**
- `new_item` (dict): The item being evaluated.

> Loads the comparison set internally via `load_listings()` (mirrors how `search_listings` reads the data file itself), so the agent call stays a single argument. The item itself is excluded from its own comparison set by `id`.

**What it returns:**
A string `str` providing a price assessment and reasoning (e.g., the item's price vs. the average/median of comparable items, and whether it reads as a deal, fair, or high).

**What happens if it fails or returns nothing:**
If no comparables exist, returns `"Insufficient data to assess price."` This is **non-fatal** — the planning loop continues and `create_fit_card` simply omits the price note.

---

### Tool 5: check_trends (Stretch Feature)

**What it does:**
Surfaces what styles are currently popular for the selected item's `category` and `style_tags`, using a small **mock platform-trends data source** (a static dict of trending tags per category, as if scraped from Depop/Poshmark) with an LLM call to phrase the summary. The returned text is fed into `suggest_outfit` so the trend visibly influences the outfit.

**Input parameters:**
- `category` (str): The category of the selected item.
- `style_tags` (list[str] | None = None): The selected item's style tags, used to pick the most relevant trend note.

**What it returns:**
A string `str` summarizing current trends relevant to the item.

**What happens if it fails or returns nothing:**
If no trend data matches, returns an empty string `""`. This is **non-fatal** — `suggest_outfit` proceeds with `trends=None` and omits trend context.

## Planning Loop

**How does your agent decide which tool to call next?**
The agent uses a sequential planning loop with conditional branches. State checked between steps drives each decision (empty `search_results` → retry branch; still empty → error branch; non-fatal stretch tools degrade to `None` rather than halting).

1. **Load profile *(Stretch)*:** Read `data/style_profile.json` if it exists; store its `preferences` text in `session["style_profile"]` (else `None`).
2. **Parse:** Extract `description`, `size`, `max_price` from the query into `session["parsed"]`.
3. **Search:** Call `search_listings(description, size, max_price)` → `session["search_results"]`.
4. **Retry with fallback *(Stretch)* — deterministic order:** If `search_results == []`, retry in this fixed sequence, stopping at the first non-empty result:
   1. drop `max_price` (keep `size`),
   2. else drop `size` (keep `max_price`),
   3. else drop both.
   On a successful retry, set `session["fallback_used"] = True` and write a human-readable `session["warning"]` naming exactly which constraint was relaxed. If all retries still return `[]`, set `session["error"]` and **return early** (skip all downstream tools).
5. **Select:** `session["selected_item"] = search_results[0]`.
6. **Price check *(Stretch, non-fatal)*:** Call `compare_price(selected_item)` → `session["price_assessment"]`.
7. **Trends *(Stretch, non-fatal)*:** Call `check_trends(category, style_tags)` → `session["trends"]` (empty string treated as `None` downstream).
8. **Suggest:** Call `suggest_outfit(selected_item, wardrobe, trends, style_profile)` → `session["outfit_suggestion"]`.
9. **Caption:** Call `create_fit_card(outfit_suggestion, selected_item, price_assessment)` → `session["fit_card"]`.
10. **Save profile *(Stretch)*:** Persist any newly parsed style preferences to `data/style_profile.json` (see State Management for the merge rule).

## State Management

**How does information from one tool get passed to the next?**
A single `session` dictionary is the single source of truth for one interaction. Each step reads the keys it needs and writes its output back, so the *same* objects flow forward without the user re-entering anything (e.g., `search_results[0]` becomes `selected_item`, which is passed into `suggest_outfit`; the returned `outfit_suggestion` is passed into `create_fit_card`).

**Canonical session keys** (used consistently here, in the diagram, and in `agent.py`):

| Key | Type | Written by | Notes |
|-----|------|-----------|-------|
| `query` | str | init | original user text |
| `parsed` | dict | step 2 | `{description, size, max_price}` |
| `search_results` | list[dict] | step 3/4 | matching listings |
| `selected_item` | dict \| None | step 5 | `search_results[0]` |
| `wardrobe` | dict | init | user's wardrobe |
| `style_profile` | str \| None | step 1 | *(stretch)* loaded preference text |
| `fallback_used` | bool | step 4 | *(stretch)* True if constraints relaxed |
| `warning` | str \| None | step 4 | *(stretch)* human-readable note on what was relaxed |
| `price_assessment` | str \| None | step 6 | *(stretch)* from `compare_price` |
| `trends` | str \| None | step 7 | *(stretch)* from `check_trends` |
| `outfit_suggestion` | str \| None | step 8 | from `suggest_outfit` |
| `fit_card` | str \| None | step 9 | from `create_fit_card` |
| `error` | str \| None | step 4 | `None` on success; set → other outputs stay `None` |

**Cross-session persistence *(Stretch — Style Profile Memory)*:** `data/style_profile.json` holds a single object, `{"preferences": "<free-text style notes>"}`. On load (step 1) its `preferences` string is placed in `session["style_profile"]` and passed to `suggest_outfit`. On save (step 10), newly detected preference phrases from the query (e.g., "I mostly wear baggy jeans and chunky sneakers") are extracted and **merged** into the existing `preferences` text (append-if-new, dedup on phrase) so prior sessions are not overwritten. The file is git-ignored (user-generated state).

## Error Handling

For each tool, describe the specific failure mode you're handling and what the agent does in response.

| Tool | Failure mode | Agent response |
|------|-------------|----------------|
| search_listings | No results match the query | Agent runs the deterministic retry (drop `max_price` → drop `size` → drop both), sets `fallback_used`/`warning` on success. If all retries still return `[]`, sets `session["error"]` with a message naming the query and suggesting what to loosen, and returns early. |
| suggest_outfit | Wardrobe is empty | The prompt instructs the LLM to give general styling advice for the item rather than referencing specific wardrobe pieces; still returns a non-empty string. |
| create_fit_card | Outfit input is missing/empty | Generates a caption focused on `new_item` alone (name, price, platform); never raises. |
| compare_price | No comparable items | Returns `"Insufficient data to assess price."`; **non-fatal** — loop continues, `price_assessment` note is simply omitted from the fit card. |
| check_trends | No matching trend data | Returns `""`; **non-fatal** — `suggest_outfit` proceeds with `trends=None`. |

## A Complete Interaction (Step by Step)

Write out what a full user interaction looks like from start to finish — tool call by tool call. Use a specific example query.

**Example user query:** "I'm looking for a vintage graphic tee under $10. I mostly wear baggy jeans and chunky sneakers. What's out there and how would I style it?"

**Step 1:** The agent loads `data/style_profile.json` (empty on a first-ever run) into `session["style_profile"]`. It parses the query into `session["parsed"] = {description: "vintage graphic tee", size: None, max_price: 10.0}`. It calls `search_listings("vintage graphic tee", None, 10.0)`.

**Step 2:** `search_listings` returns `[]` — the cheapest listing in the dataset is $12, so nothing matches under $10. The **Retry Logic** activates and drops `max_price` first; the retry succeeds, with the top keyword match being **"Graphic Tee — 2003 Tour Bootleg Style" ($24, size L)**. The agent sets `fallback_used = True` and `warning = "No matches under $10 — showing results without the price limit."`

**Step 3:** `compare_price(selected_item)` loads the listings, compares against other `tops`/tee-tagged items (e.g., Y2K Baby Tee $18, Vintage Band Tee $19), and reports something like "$24 is on the higher end for vintage tees (comps average ~$20)."

**Step 4:** `check_trends("tops", ["vintage", "graphic"])` returns a trend note such as "Bootleg-style band/graphic tees are trending right now." (`session["trends"]`).

**Step 5:** `suggest_outfit(selected_item, wardrobe, trends, style_profile)`. The LLM combines the tee with wardrobe pieces (and the relaxed-budget context), influenced by the trend note.

**Step 6:** `create_fit_card(outfit_suggestion, selected_item, price_assessment)` produces the shareable caption, weaving in the price note.

**Step 7:** The agent merges the newly detected preferences ("baggy jeans", "chunky sneakers") into `data/style_profile.json` so the next session reuses them without re-entry, then returns the session.

**Final output to user:**
The user sees the top listing (Graphic Tee — 2003 Tour Bootleg Style, $24) with the relaxed-price warning, the price-fairness note, the outfit idea built around their saved baggy jeans and chunky sneakers (influenced by the bootleg-tee trend), and a shareable fit card caption.
